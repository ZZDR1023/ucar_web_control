## 本次已完成的关键修复

### 1. 独立相机桥

新增/使用：

```text
web_panel/ucar_camera_bridge.py
```

作用：

- 用 OpenCV 直接读 `/dev/video0`。
- 发布 ROS 话题 `/ucar_camera/image_raw`。
- 日志写到 `/tmp/ucar_camera_bridge.log`。
- 不再依赖原厂 `ucar_camera.py`，因为原厂节点可能“进程和话题存在，但没有新图像”。

相机桥启动命令改为直接 `nohup python3 -u`，不要套内层 `bash -lc`：

```bash
nohup python3 -u /home/ucar/web_panel/ucar_camera_bridge.py _device:=/dev/video0 _topic:=/ucar_camera/image_raw _width:=1280 _height:=720 _rate:=10 >/tmp/ucar_camera_bridge.log 2>&1 < /dev/null &
```

### 2. 硬释放 `/dev/video0`

Web 后端启动人体跟随前，`release_web_camera_for_person_follow()` 现在会：

- 清零 Web camera client。
- 停止视觉巡线占用。
- 调用 `hard_release_video_device_for_person_follow()`。
- 用 `fuser /dev/video0 /dev/ucar_video` 找占用 PID。
- 对非 Web 后端自身的占用者执行 `kill -9`。
- `pkill -9` 清旧 `ucar_camera.py` / `ucar_camera_bridge.py`。
- `time.sleep(1.5)` 等 Linux 内核释放 USB 视频节点。

原因：用户确认硬杀逻辑生效，`ucar_camera_bridge` 已经能成功运行并发布。

### 3. 相机就绪判定改为完整 Image

之前错误点：

- Web 后端只检测 `/ucar_camera/image_raw/header`。
- 用户反馈相机桥已在发布，但 Web 仍报“未就绪”。

当前修复：

```python
rospy.wait_for_message("/ucar_camera/image_raw", Image, timeout=10.0)
```

要求：

- 不再检测 `/header` 子字段。
- 必须收到完整 `sensor_msgs/Image` 才判定相机就绪。
- 超时时间从 5 秒改为 10 秒。

### 4. 相机桥读帧失败退出

`web_panel/ucar_camera_bridge.py` 当前逻辑：

- `cap.read()` 连续 30 帧失败或返回 `None`。
- 调用 `rospy.logerr(...)` 打明显错误。
- `raise RuntimeError(...)` 退出进程。

目的：不能占着 `/dev/video0` 不发数据也不报错。

### 5. Web 控制延迟修复

之前问题：

- 做人体跟随后，Web 方向键、松手停止、`cmd_vel`、`manual_stop` 请求变成 6-18 秒。

根因：

- `/api/cmd_vel` 和 `/api/manual_stop` 每次都无条件执行人体跟随清理。
- 即使人体跟随已经停止，也反复 `pkill`、杀相机桥、查进程。

当前修复：

- 普通方向键/松手停止只在缓存显示人体跟随仍运行时才清理。
- STOP、显式“停止人体跟随”、导航/巡逻/语音等仍会强制停止人体跟随。
- 人体跟随状态加短缓存，避免状态轮询每次 `pgrep`。

实测曾恢复到：

- `/api/manual_stop` 约 `0.10s`
- `/api/cmd_vel` 零速度约 `0.10s`
- `/api/status` 缓存后约 `0.08s`
- `/api/person_follow/status` 缓存后约 `0.05s`

用户要求：“现在总是显示人体跟随服务未就绪，或者是前方有障碍，先把有障碍停止移动去掉。”

## 代码审查后的本地补丁

2026-05-14 末尾又做了一次本地代码审查，发现并补了 3 个点。注意：这 3 个点因为小车随后仍然 `No route to host`，尚未部署到小车，只在本地仓库通过测试。

### 1. 巡线记录器也要停止

问题：

- `release_web_camera_for_person_follow()` 之前只清零 Web 预览和停巡线控制。
- 如果 `line_follow_recorder` 仍 active，`camera_loop()` 仍会认为需要摄像头，Web 后端自身会继续占用 `/dev/video0`。
- 硬杀逻辑会跳过当前 Web 后端 PID，因此设备可能仍没释放。

本地修复：

- 如果 recorder active，调用 `line_follow_recorder.stop()`。

### 2. 相机未就绪时不启动原厂人体跟随

问题：

- 之前即使 10 秒内没有收到完整 `/ucar_camera/image_raw`，代码仍会继续启动原厂 `person_follow.py`。
- 这会留下“相机未就绪但人体跟随节点已启动”的残留状态。

本地修复：

- `camera_ready = wait_for_camera_message(timeout=10.0)` 后，如果失败：
  - 停止相机桥；
  - 设置状态为“相机话题未就绪”；
  - 直接返回失败；
  - 不启动 `PERSON_FOLLOW_COMMAND`。

### 3. 服务成功必须包含 start_follow

问题：

- `start_person_follow_services()` 之前只用 `detect["returncode"] == 0` 判定成功。
- 如果 `/person_follow/start_person_follow` 调用失败，Web 仍可能误报成功。

本地修复：

- `wait_for_person_follow_ready()` 同时等待：
  - `/person_follow/start_person_detect`
  - `/person_follow/start_person_follow`
- `start_person_follow_services()` 返回成功必须满足：
  - `detect["returncode"] == 0`
  - `follow["returncode"] == 0`
